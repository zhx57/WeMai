# YUbot 原生语音发送实现评估报告

## 结论先行

YUbot 的方案在机制上确实可以产生微信原生语音条。它不是调用隐藏的 `SendVoice` API，也不是把 WAV/SILK 文件直接注入微信，而是：

> 让微信 4.x 正常进入实时录音模式，再通过虚拟声卡或“立体声混音”把预制 WAV 播放成微信的麦克风输入，最后点击微信的录音发送按钮。

因此：

- “原生语音条”是真的：消息由微信自身录音流程创建。
- “一定有正确声音”则不能保证：现有验证只确认出现了新的绿色气泡，不验证气泡里的音频内容。
- WeMai 当前绑定微信 3.9.11.17，不能直接移植。3.9.x 没有 PC 端原生语音消息录制按钮。
- 最现实路线是升级到微信 4.1.9 或经过实机验证的更高 4.x，并重做/替换 wxauto 3.x 的界面适配，然后复用 YUbot 的音频路由思路。

总体判断：方案可行，但属于对桌面环境高度敏感的 UI 自动化工程，不是稳定的协议级语音发送能力。

# 问题 1：YUbot 的 `send_voice` 是怎么工作的？

核心实现位于 [pc_weixin_bridge.py](/workspace/robot_analysis/YUbot/src/wechat_agent_adapter/pc_weixin_bridge.py:3609)。

## 1. 完整流程

`send_voice()` 的顺序如下：

1. 获取全局发送锁，防止同一桥接实例并发操作微信。
2. 检查音频文件，读取 WAV 时长。
3. 计算微信录制时长：
   - 有 WAV 时：`音频时长 + 0.25 秒`
   - 最少 1.2 秒
   - 最多 55 秒
4. 枚举 Windows 音频端点，寻找：
   - Stereo Mix / 立体声混音
   - VB-CABLE
   - VoiceMeeter
   - 其他名称匹配的虚拟声卡
5. 激活微信窗口，确认登录状态和目标聊天。
6. 截取发送前的微信窗口图像。
7. 临时切换 Windows 音频路由。
8. 提前解码、重采样并“预热”音频播放。
9. 坐标点击微信的“发语音”按钮，开始录音。
10. 等待 80 毫秒。
11. 将 WAV 播放到虚拟声卡输入端或系统默认输出。
12. 等待完整音频时长。
13. 坐标点击绿色发送按钮。
14. 停止播放，并恢复原来的 Windows 默认录音/播放设备。
15. 截取发送后的窗口图像，检查是否出现新的右侧绿色气泡。
16. 只有检测到新气泡才返回成功，否则报错且不确认任务完成。

## 2. 它怎么切换音频路由？

音频路由实现在 [audio_routing.py](/workspace/robot_analysis/YUbot/src/wechat_agent_adapter/audio_routing.py:782)。

它同时使用了两种 Windows 机制。

### 注册表用于枚举和识别设备

代码读取：

```text
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\MMDevices\Audio
```

从注册表获取设备名称、友好名称、驱动、Provider、设备状态和 endpoint ID。

它通过名称关键词识别候选设备：

- 捕获端：
  - `cable output`
  - `vb-audio`
  - `voicemeeter output`
  - `virtual audio`
  - `stereo mix`
  - `立体声混音`
- 播放端：
  - `cable input`
  - `voicemeeter input`
  - `virtual audio`

这里容易混淆，但 VB-CABLE 的命名逻辑是：

```text
程序播放到 CABLE Input
              ↓
微信从 CABLE Output 录音
```

### Core Audio API 用于修改默认设备

它不是直接修改注册表里的“默认设备”值，而是通过 Windows 未公开但广泛使用的 `IPolicyConfig.SetDefaultEndpoint`，临时设置三个角色的默认端点：

- Console
- Multimedia
- Communications

`begin_native_voice_audio_route()` 会保存：

- 原默认录音设备
- 原默认播放设备

发送结束后，`restore_native_voice_audio_route()` 再逐角色恢复，并验证恢复结果。

### 两类路由

#### 虚拟声卡

对于 VB-CABLE/VoiceMeeter：

- Windows 默认录音设备切到虚拟声卡的 capture 端，即 `CABLE Output`
- WAV 通过 PortAudio 定向播放到 `CABLE Input`
- 不切换系统默认播放设备

这是比较合理的设计，因为其他系统声音继续走耳机/扬声器，不会一起灌入语音条。

#### 立体声混音

对于 Stereo Mix：

- 默认录音设备切到 Stereo Mix
- 默认播放设备可能临时切到同一物理声卡的 render 端
- WAV 用 `winsound` 播到系统默认输出
- Stereo Mix 捕获该声音并交给微信

这个模式更容易混入提示音、视频和其他系统声音。

## 3. 它怎么让微信录到预制音频？

本质是音频回环：

```text
预制 WAV
   ↓
winsound / PortAudio
   ↓
虚拟播放端 CABLE Input
   ↓
虚拟声卡内部转发
   ↓
虚拟录音端 CABLE Output
   ↓
微信当前麦克风
   ↓
微信原生实时录音
   ↓
原生语音气泡
```

微信并不知道输入来自文件。对微信而言，它只是从一个普通的 Windows 麦克风端点实时采集声音。

这也意味着发送必须实时进行：30 秒音频至少占用约 30 秒。

## 4. 怎么点击录音和发送按钮？

不是 UIA 控件调用，而是相对于微信窗口矩形的比例坐标点击。

相关常量位于 [pc_weixin_bridge.py](/workspace/robot_analysis/YUbot/src/wechat_agent_adapter/pc_weixin_bridge.py:61)：

```python
VOICE_START_BUTTON_X = 0.863
VOICE_START_BUTTON_Y = 0.955

VOICE_CANCEL_BUTTON_X = 0.734
VOICE_CANCEL_BUTTON_Y = 0.955

VOICE_SEND_BUTTON_X = 0.947
VOICE_SEND_BUTTON_Y = 0.955
```

代码注释明确说明这些是：

> WeChat 4.x compose-bar voice buttons

并区分了两个按钮：

- 左侧麦克风图标约在 `0.52`：语音转文字，不是语音消息
- 发送按钮旁的圆形图标约在 `0.863`：发语音

点击过程为：

1. UIA 找到微信窗口并确保前台。
2. 取得窗口的 left/top/width/height。
3. 计算绝对屏幕坐标：
   - `x = left + width × fraction`
   - `y = top + height × fraction`
4. 调用 Win32：
   - `SetCursorPos`
   - `mouse_event(MOUSEEVENTF_LEFTDOWN)`
   - `mouse_event(MOUSEEVENTF_LEFTUP)`

因此 YUbot 是混合自动化：

- UIA：找窗口、切聊天、验证输入区域和目标联系人
- Win32 坐标：点击语音录制、取消、发送按钮

一个实现缺陷是 `_click_window_fraction()` 捕获点击异常后直接 `pass`。调用方可能得到“已点击”的返回描述，实际上点击并未成功。后面的截图验证通常会发现没有新气泡，但错误定位会变得困难。

## 5. 它怎么验证语音气泡发出了？

发送前后各截一张微信窗口截图，然后调用 `outgoing_green_bubble_delta()`。

验证逻辑只扫描：

- 窗口右侧区域
- 聊天记录的下半部分
- 新出现的微信绿色像素连通区域

候选区域至少满足：

- 120 个绿色像素
- 宽度至少 24 像素
- 高度至少 14 像素
- 中心位于窗口右侧
- 位于靠近输入栏的新消息区域

检测到后，`send_voice()` 返回：

```python
"native_voice_bubble": True
"delivery_verified": "visible_outgoing_bubble"
```

### 验证能力的边界

它能证明：

- 点击后聊天区出现了一个新的右侧绿色气泡。

它不能证明：

- 气泡一定是语音类型，而不是其他绿色消息。
- 音频内容与输入 WAV 一致。
- 语音不是静音、截断或杂音。
- 服务器已经可靠投递给对方。
- 没有发送失败标记。

媒体发送另有 pending/failed 状态检测，但 `send_voice()` 使用的是较简单的绿色气泡检测，没有调用媒体投递状态验证。

所以这里更准确的含义是：

> “看起来微信创建了一条新的出站消息”，而不是端到端的语音内容验证。

## 6. 失败时怎么回滚？

`send_voice()` 使用 `try / except / finally`：

- 录音已开始但后续失败：
  - 点击取消按钮
- 播放已开始：
  - 调用 `sounddevice.stop()`
  - 调用 `winsound.PlaySound(None, 0)`
- 音频路由发生过切换：
  - 恢复原默认录音设备
  - 恢复原默认播放设备
  - 恢复因发送而临时启用的设备状态

优点是大部分失败路径都有恢复。

但仍有两个风险：

1. 恢复失败只记录在结果中，不会撤回已经发送的语音，也未必让整个发送返回失败。
2. 进程崩溃、被强杀或 Windows 卡死时，`finally` 无法执行，默认声卡可能停留在虚拟设备。

## 7. 为什么要预热播放？

相关实现在 [pc_weixin_bridge.py](/workspace/robot_analysis/YUbot/src/wechat_agent_adapter/pc_weixin_bridge.py:3994) 和 [audio_routing.py](/workspace/robot_analysis/YUbot/src/wechat_agent_adapter/audio_routing.py:1248)。

以前如果在微信开始录音后才做：

- 读取 WAV
- 转换为 float32
- 重采样
- 调整声道数
- 匹配 PortAudio 设备
- 打开音频设备

这些工作可能耗费数百毫秒，甚至随音频长度增长。微信已经开始录音，却暂时没有音频输入，于是语音条开头出现一段 dead air。

YUbot 的修正是：

1. 微信开始录音前完成 WAV 解码和重采样。
2. 把完整音频保存在内存数组中。
3. 向目标 PortAudio 设备播放约 0.25 秒静音，尝试唤醒设备和驱动。
4. 微信开始录音后，只执行一次非阻塞 `sd.play()`。
5. 再固定等待 80 毫秒后启动播放。

### 预热并非绝对保证

`prewarm_portaudio_device()` 使用的是 `sounddevice.sd.play()` 便利接口。预热播放结束后，真正播放又调用一次 `sd.play()`。它不一定复用完全相同、持续打开的 OutputStream。

所以“复用 warm stream”的注释略显乐观。它确实可以：

- 提前触发设备初始化
- 提前加载 PortAudio/驱动路径
- 提前发现设备格式错误

但不能完全消除：

- 第二次打开流的延迟
- Windows 调度延迟
- 微信麦克风开始采集的延迟
- 固定 80 毫秒造成的开头静音

`winsound` 模式也没有真正的设备流预热，只是避免了解码/重采样开销。

## 8. 最终判断：真的能发原生语音条吗？

结论是：

> 在微信 4.x 原生支持语音消息、窗口坐标准确、微信实际使用所切换的虚拟麦克风、虚拟声卡路由正确的前提下，它确实能创建微信原生语音条。

但必须区分两件事：

- 原生气泡：可信，微信自身录制并创建。
- 气泡内容正确：现有代码不能保证。

尤其要注意，`begin_native_voice_audio_route(require_content_proven=True)` 表面上要求环回验证，但如果检测到“可信”的 VB-CABLE/VoiceMeeter，它会自动跳过硬性 loopback proof，直接假设该设备必然互通。因此以下情况仍可能发送空音：

- 微信固定使用另一个麦克风，而不是 Windows 默认设备
- PortAudio 选错同名设备
- 微信在录音开始时缓存了旧设备
- 虚拟声卡采样格式或通道配置异常
- 播放端和录音端并非同一套虚拟 cable
- 播放启动延迟或播放失败但接口表面返回 started

代码没有发现“致命到完全不可用”的机制错误，但其成功判定不足以排除静音和杂音。

# 问题 2：前提条件

## 操作系统

必须是 Windows。

原因包括：

- Windows MMDevices 注册表
- Windows Core Audio `IPolicyConfig`
- `winreg`
- `winsound`
- Win32 鼠标事件
- Windows UI Automation
- 微信 Windows 客户端

macOS 和 Linux 不能直接使用这个模块。

## 微信版本

必须是带有 PC 原生“发送语音消息”功能的微信 4.x。

结合既有调研，明确的最低版本是：

> 微信 Windows 4.1.9

微信 4.1.8 的语音输入是语音转文字，不等于语音条。3.9.x 也没有这里所点击的录音按钮。

此外，YUbot 的默认坐标是 2026-06-16 在某个实际 4.x 版本和布局上测量得到的，并不意味着所有 4.x 都能直接使用同一组坐标。

## 硬件和驱动

至少需要以下一种：

1. 推荐：VB-CABLE 或 VoiceMeeter 等真正的虚拟声卡。
2. 次选：物理声卡提供并启用了 Stereo Mix/立体声混音。

Python 播放路径还需要：

- `sounddevice`
- `numpy`
- 可用的 PortAudio backend

虚拟声卡比 Stereo Mix 更适合生产环境，因为它可以只把目标 WAV 注入微信，不混入系统其他声音。

## 微信麦克风设置

理想配置是：

- 微信麦克风使用“系统默认设备”
- YUbot 临时把 Windows 默认录音设备切到 `CABLE Output`

如果微信设置中固定选择了物理麦克风，修改 Windows 默认端点可能不生效。这时必须：

- 在微信设置中直接选 `CABLE Output`，或
- 把微信改回“系统默认”，并验证它能动态跟随默认设备变化

还需要实测微信是否在每次开始录音时重新打开默认设备。如果微信启动时就缓存设备，可能需要重启微信或重新选择麦克风。

## 前台和焦点

要求较严格：

- 微信窗口必须存在且已登录。
- 必须打开正确聊天。
- 微信必须能被切到前台。
- 录音开始和发送时窗口尺寸、布局不能变化。
- 输入栏不能被弹窗、菜单、提示框或其他窗口遮挡。
- 鼠标不能同时被用户或另一自动化程序抢夺。
- 录音过程中不应切换聊天或操作微信。
- 屏幕不能锁定；远程桌面断开或会话切换也可能影响 UIA、截图和坐标点击。

# 问题 3：WeMai 当前能直接用吗？

## 明确结论

> 不能直接移植；需要较大范围的版本和 UI 适配，但不是完全不可行。

## 原因一：WeMai 当前绑定微信 3.9.11.17

[wxauto.py](/workspace/WeMai-fix/wxauto/wxauto.py:24) 明确写着：

```python
VERSION = "3.9.11.17"
```

虽然当前代码把版本不匹配改成了警告并继续运行，但这只绕过了版本门禁，不代表 wxauto 的控件树解析可以兼容微信 4.x。

## 原因二：微信 3.9.x 没有该原生语音按钮

YUbot 依赖微信输入栏上的：

- 发语音入口
- 录音模式取消按钮
- 录音模式绿色发送按钮

PC 微信 3.9.x 没有这一套原生录制语音消息 UI。没有录音入口，虚拟声卡即使配置正确，也没有微信录音流程可以接收它。

因此不能通过“给现有 wxauto 加一个 `SendVoice` 方法”解决。

## 原因三：WeMai 当前语音段直接降级为文本

[wx_Processer.py](/workspace/WeMai-fix/wx_Processer.py:275) 对 `voice` 段的处理是：

```python
data = "[语音消息]"
```

随后按文本发送。

发送线程 [wx_Listener.py](/workspace/WeMai-fix/wx_Listener.py:209) 仅允许：

```python
{"text", "image", "file"}
```

当前架构里没有：

- `voice` 出站类型
- WAV 路径/时长处理
- 音频路由生命周期
- 录音状态控制
- 语音气泡验证

## 如果要采用 YUbot 方案，需要先做什么？

第一步不是复制 `send_voice()`，而是：

1. 升级微信到 4.1.9 或经过实机验证的 4.x。
2. 解决 wxauto 3.x 对微信 4.x 的兼容问题：
   - 换用支持微信 4.x 的自动化库，或
   - 重新适配主窗口、会话列表、聊天窗口和输入框控件树，或
   - 将 YUbot 的 PC 微信 4.x bridge 作为独立发送后端接入 WeMai。
3. 实机标定语音按钮坐标。
4. 安装并验证虚拟声卡。
5. 才能增加 WeMai 的 `voice` 发送链路。

单纯升级微信但继续依赖当前 `wxauto.ChatWnd` 的 3.x 控件结构，风险很高。

# 问题 4：最现实的移植路线

## 推荐路线

> 保留 WeMai 的消息处理和发送队列，把“微信 4.x 出站驱动”替换或扩展为独立 bridge；复用 YUbot 的音频路由逻辑和坐标录音流程。

## 1. 微信和 wxauto 版本

建议最低使用微信 Windows 4.1.9。

更稳妥的做法不是追求“最新 4.x”，而是固定一个经测试的具体版本，验证：

- 原生发语音按钮存在
- 控件树稳定
- 坐标稳定
- 微信能跟随默认麦克风
- UI 自动化不被版本更新破坏

现有 wxauto 3.9.11.17 不应仅靠跳过版本检查继续使用。可选择：

- 找到明确支持微信 4.x 的 wxauto/自动化分支；或
- 将 YUbot 的 `pc_weixin_bridge` 用作微信 4.x 的会话导航和发送驱动；或
- 对 WeMai 当前 wxauto 控件树进行 4.x 重适配

从已有代码看，直接接入一个独立 bridge 通常比把 wxauto 3.x 全面改成 4.x 更可控。

## 2. `audio_routing.py` 能否直接复用？

技术上大部分可以复用。

它相对独立，涵盖：

- 注册表枚举音频端点
- 虚拟声卡识别
- 默认 capture/render 切换
- 原设备恢复
- PortAudio 设备匹配
- WAV 解码、重采样和播放
- loopback 诊断

但不是复制单个文件就完整可用：

- `prepare_native_voice_audio_playback()` 等编排函数在 `pc_weixin_bridge.py`，不全在 `audio_routing.py`
- 依赖 Windows、`sounddevice`、`numpy`
- 需要处理日志、状态文件和异常结构
- 需要确认 YUbot 的许可证是否允许直接复制
- 建议抽成 WeMai 独立的 `native_voice_audio` 服务，而不是塞入 wxauto 控件类

复用难度：中等。

## 3. UIA 和坐标点击怎么结合？

两者不冲突，YUbot 本身就是混合方式。

推荐分工：

- UIA：
  - 定位微信窗口
  - 切换目标聊天
  - 验证联系人
  - 获取窗口矩形
  - 激活窗口
- 坐标：
  - 点击“发语音”
  - 点击取消
  - 点击录音发送
- 截图：
  - 验证录音模式确实出现
  - 验证新语音气泡出现

不要直接复用 YUbot 的全局默认比例，而应：

1. 固定测试微信版本。
2. 分主窗口和独立 `ChatWnd` 分别标定。
3. 考虑窗口最小尺寸、最大化状态和 DPI。
4. 优先尝试从 UIA 树中识别按钮；识别不到才坐标点击。
5. 点击开始后先验证 UI 已进入录音模式，再开始播放，避免把 WAV 播完却根本没在录音。
6. 点击发送前再次验证仍处于录音状态。

这是比原实现更重要的增强点。

## 4. WeMai 侧需要增加的链路

大致包括：

```text
voice segment
    ↓
解析本地 WAV / 下载并转 WAV
    ↓
进入现有单线程微信发送队列
    ↓
锁定目标聊天
    ↓
建立虚拟音频路由
    ↓
开始微信录音
    ↓
实时播放 WAV
    ↓
发送并验证气泡
    ↓
恢复音频设备
```

具体要修改的概念区域：

- Processor：不再把 voice 转成 `[语音消息]`
- Listener：允许 `kind == "voice"`
- 发送队列：为语音发送保留独占时间
- 微信驱动：增加 `SendVoice`
- 配置：虚拟声卡、坐标、最大时长、loopback 检测
- 诊断：设备路由自检
- 运维：进程退出时恢复设备

## 5. 工程量

如果只做单机 PoC：

> 中等改造

前提是使用 YUbot bridge 或已有可工作的微信 4.x 自动化层，预计重点在音频模块接入、按钮标定和 WeMai 出站类型扩展。

如果要求生产级稳定性，包括：

- 多 DPI
- 多窗口尺寸
- 断线恢复
- 微信升级兼容
- 空音检测
- 设备恢复
- 长时间运行
- 用户同时操作微信
- 多种虚拟声卡

则属于：

> 大改造

真正困难的不是播放 WAV，而是确保每一次都录到正确内容、发给正确联系人，并在失败后恢复桌面环境。

## 6. 风险评估

| 风险 | 等级 | 说明 |
|---|---:|---|
| 微信 4.x UI/控件树兼容 | 高 | 当前 wxauto 针对 3.9.11.17 |
| 坐标随版本变化 | 高 | 按钮右对齐且受窗口尺寸、DPI、布局影响 |
| 发送空音 | 中高 | 气泡验证不验证音频内容 |
| 虚拟声卡部署 | 中高 | 驱动安装、权限、设备重名、采样率 |
| 默认设备恢复失败 | 中 | 崩溃或 Core Audio 调用失败可能留下错误默认设备 |
| 并发和用户干扰 | 高 | 录音期间必须独占微信窗口和鼠标 |
| 微信更新 | 高 | 一次小更新就可能改变按钮位置或窗口结构 |
| 账号风险 | 低到中 | 属于模拟人工 UI，通常低于 Hook/DLL 注入，但高频自动化仍可能触发风控 |
| 实时效率 | 高 | 语音时长与发送耗时基本 1:1 |

# 问题 5：缺陷和坑

## 1. 必须实时发送

发送耗时约为：

```text
WAV 时长 + 路由切换 + UI 等待 + 发送确认
```

60 秒音频不能瞬间发送。

而且代码最大录制时间为 55 秒，超过约 54.75 秒的 WAV 会因为上限而被截断。这个限制必须在 WeMai 上游显式校验或切段，不能静默截断。

## 2. 虚拟声卡依赖强

可能出现：

- 驱动未安装
- 设备被禁用
- 多个同名 VB-CABLE
- Windows endpoint ID 变化
- PortAudio 枚举顺序变化
- 微信固定使用物理麦克风
- 采样率不匹配
- 远程桌面改变默认音频端点

关键词匹配也可能误判。“名称看起来像虚拟声卡”并不等于输入输出一定是同一条 cable。

## 3. “可信虚拟声卡”会跳过内容证明

代码对 VB-CABLE/VoiceMeeter 默认认为必然互通，并允许跳过 `playrec` 环回硬证明。

这解决了 PortAudio `-9997` 等诊断误报，但代价是：

> `require_content_proven=True` 并不总是真的完成了 content proof。

所以生产部署仍应在初始化时运行一次完整环回诊断，并人工试听诊断录音。

## 4. 坐标脆弱

比例坐标比固定像素好，但仍受以下因素影响：

- DPI 缩放
- 微信窗口宽度
- 输入框高度
- 系统字体缩放
- 微信主题
- 工具栏图标增减
- 独立聊天窗口与主窗口差异
- 4.x 小版本改版
- 录音模式未成功切换

代码注释自己也承认：工具栏右对齐会使比例随窗口宽度和 DPI 漂移。

## 5. 点击失败被吞掉

`_click_window_fraction()` 捕获异常后直接忽略，调用方仍会记录类似：

```text
voice_start_coordinate:0.863,0.955
```

这只是“尝试了这个坐标”，不是“点击成功”。

更可靠的实现应在每次点击后验证界面状态。

## 6. 并发能力差

YUbot 的实例锁只能防止同一对象内并发，不能防止：

- 用户操作鼠标
- 其他自动化进程操作微信
- 系统通知抢焦点
- 多进程同时发送
- 另一个程序更改默认声卡

语音发送期间整个微信前台和音频路由都应视为全局独占资源。

## 7. Dead air 只被缓解，没有完全解决

已有改进是合理的：

- 提前解码和重采样
- 提前匹配设备
- 播放短静音预热
- 微信开始录音后快速触发内存缓冲播放

但仍可能有：

- 固定 80ms 空白
- 微信录音启动延迟
- 第二次 `sd.play()` 重新开流
- `winsound` 初始化延迟
- 虚拟驱动缓冲延迟
- 音频开头被微信降噪/自动增益削掉

如果需要高质量结果，建议在 WAV 前后主动加入少量静音，并通过接收端实测确定合适值，而不是假设 80ms 对所有机器都正确。

## 8. 截图验证可靠性有限

当前绿色气泡验证容易出现两类问题。

假阳性：

- 其他程序或用户同时发出一条文本消息
- 旧绿色气泡因滚动位置变化形成新像素区域
- 界面动画或布局变化产生绿色连通块

假阴性：

- 微信主题或颜色调整
- 气泡尺寸较小
- 发送后位置不在预期下半区域
- 截图时动画尚未完成
- DPI/缩放导致像素阈值不适合

它也没有识别语音气泡的波形、时长或播放控件。

## 9. 不验证音频内容

这是最大的功能性缺口。

现有链路没有：

- 发送后自动播放语音气泡并录制比对
- 从另一账号接收后计算音频相似度
- 检测气泡是否静音
- 检测音频是否被截断
- 检测开头/结尾 dead air 长度

所以它能证明 UI 成功，不能证明音频成功。

## 10. 音频格式有限

发送路径当前只支持 WAV。其他格式必须预先转换。

对于不合法 WAV，时长可能读成 0，随后走默认录制时长；准备播放阶段通常会拒绝，但上游最好先统一转成明确格式，例如 PCM WAV。

## 11. 恢复机制不是崩溃安全的

正常异常可以进入 `finally`，但以下情况无法保证恢复：

- 进程被强制终止
- Python 崩溃
- 系统关机
- 微信或音频驱动卡死
- 服务管理器直接 kill

生产化需要启动时检查“是否遗留在虚拟默认设备”，并提供独立恢复命令或 watchdog。

# 最终建议

建议把 YUbot 定位为“已经验证了技术路线”的参考实现，而不是可以直接复制到 WeMai 的即插即用模块。

推荐实施顺序：

1. 在独立测试机安装微信 4.1.9 或选定的固定 4.x。
2. 安装 VB-CABLE。
3. 手工确认微信能从 `CABLE Output` 录到播放进 `CABLE Input` 的 WAV。
4. 验证 YUbot 的 `send_voice()` 在该机器能发出有声语音条。
5. 确定微信 4.x 自动化层：优先复用/抽取 YUbot bridge，而不是强行让 wxauto 3.x 跨大版本运行。
6. 将 WeMai 的 voice segment、发送队列和 bridge 接通。
7. 增加录音状态验证、音频环回自检和崩溃恢复。
8. 固定微信版本、窗口布局和 DPI 后再投入长期运行。

一句话总结：

> WeMai 可以借鉴 YUbot 发原生语音，但必须先升级到微信 4.1.9+并更换或重适配微信 4.x 自动化层；最现实方案是复用 YUbot 的虚拟声卡路由和坐标录音 bridge，而不是在现有微信 3.9.11.17 的 wxauto 上直接加接口。
